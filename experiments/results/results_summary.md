# 实验结果总结

> 实验时间: 2026-06-22
> 实验环境: Ollama + Qwen2.5-7B (本地) / DeepSeek API (云端) / Qwen2.5-1.5B (SLM 裁判)

---

## 实验一：延迟性能测试

### 汇总数据

| 指标 | Mode A (本地直答) | Mode B/C (云端) | 差异 |
|------|:---:|:---:|:---:|
| **平均 TTFT** | 6,908 ms | 22,047 ms | 云端慢 3.2x |
| **平均 TPOT** | 27.38 ms | 27.49 ms | 基本持平 |

| ID | 场景 | 模式 | TTFT (ms) | TPOT (ms) | 总延迟 (ms) | 输出分片 |

### 分析

1. **TTFT 差异显著** — 本地平均 6.9s vs 云端 22.0s。云端 TTFT 高是因为包含网络往返 + DeepSeek API 排队时间。本地 Ollama 首次加载模型后后续请求更快。

2. **TPOT 基本一致** — 两者都在 27ms/token 左右。这说明：
   - 本地 Qwen2.5-7B 的 token 生成速度与 DeepSeek API 接近
   - 一旦开始出 token，流式体验相似

3. **总延迟差异主要由 TTFT 决定** — 本地平均 8.9s vs 云端 52.8s。云端任务本身更复杂（输出更长），所以总延迟差距比 TTFT 更大。

4. **Mode C (脱敏上云) 延迟介于两者之间** — b3/b4 的 TTFT (13.9s/7.8s) 低于纯云端 b1/b2 (33.1s/26.5s)，因为脱敏后的查询更短，云端处理更快。

## 实验二：Token 节省量测试

### 汇总数据

| 指标 | 数值 |
|------|------|
| 总场景数 | 15 |
| 路由到本地 | 11 个 (73.3%) |
| 路由到云端 | 4 个 (26.7%) |
| 纯云方案总成本 | CNY 0.0165 |
| 我们的方案成本 | CNY 0.0133 |
| **节省金额** | **CNY 0.0032** |
| **节省比例** | **19.3%** |

| s15 | 中等知识 | 本地 | A | 20/542 | 0.0011 | 0 | 0.0011 |

### 分析

1. **73% 的日常请求被路由到本地** — 15 个场景中 11 个走本地 LLM，包括简单问答、计算、时间查询，以及含敏感信息但低复杂度的任务。

2. **节省比例 19.3%** — 看起来不高，原因是 4 个云端请求的输出很长（每个 1000-2200 tokens），占了总 token 的大头。如果按请求次数算，73% 的请求免费。

3. **敏感信息 + 低复杂度 = 本地处理** — s12（含手机号）、s13（含银行卡）、s14（含联系方式）都走了本地，因为复杂度低（L1-L2），不需要云端大模型。

4. **敏感信息 + 高复杂度 = 脱敏上云** — s09（含邮箱的复杂分析）、s11（含邮箱的企业分析）走了 Mode C，脱敏后发给云端。

5. **成本主要由长输出决定** — 云端 4 个请求的输出 tokens 合计 ~6,600，占总输出的 75%+。短问答类的成本极低。

### 扩展推算

如果一个用户每天使用 50 次，其中 70% 走本地（35 次），30% 走云端（15 次），按本次实验的平均成本：

```text
纯云方案: 50 次 × (CNY 0.0165 / 15) ≈ CNY 0.055/天 ≈ CNY 1.65/月
我们的方案: 15 次 × (CNY 0.0133 / 15) ≈ CNY 0.0133/天 ≈ CNY 0.40/月
月节省: CNY 1.25 (约 76%)
```

## 总结

| 维度 | 结论 |
|------|------|
| **延迟** | 本地 TTFT 比云端快 3.2x，总延迟快 5.9x |
| **TPOT** | 本地与云端基本持平（~27ms/token） |
| **成本** | 73% 的请求免费处理，整体节省 19.3% 的 token 成本 |
| **隐私** | 敏感信息（手机号/银行卡/邮箱）在低复杂度场景下完全不出本地 |
| **适用场景** | 日常简单问答、知识查询、计算、时间等占大多数使用场景的任务全部本地处理 |

核心价值：**用一个本地 7B 模型处理 70%+ 的日常请求，既保护隐私又节省成本，同时对复杂任务保持云端大模型的质量。**
